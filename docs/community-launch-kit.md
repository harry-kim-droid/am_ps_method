# 커뮤니티 출시 배포 키트

홍보 링크: `https://harry-kim-droid.github.io/am_ps_method/`

커뮤니티마다 같은 글을 복사하지 말고, 실제 경험이나 질문에 맞춰 첫 두 문장을 바꾼다. 광고성 게시물을 금지하는 곳에는 올리지 않는다.

## GeekNews / 개발 커뮤니티용

### 제목

혼자 개발할 때 PR 코드리뷰 공백을 Codex Action으로 메워봤습니다

### 본문

1인 개발에서는 PR을 만들어도 실제로 두 번째 사람이 봐주지는 않습니다. 그래서 PR이 열리면 Codex가 행동 버그, 테스트 누락, 호환성 위험만 먼저 확인하도록 GitHub Actions를 구성했습니다.

처음에는 댓글이 많을수록 좋다고 생각했는데, 실제로는 스타일 지적과 근거 없는 추측을 버리는 것이 더 중요했습니다. 지금 버전은 읽기 전용으로 실행하고, 댓글 작성 권한도 별도 job으로 분리했습니다.

설정 과정과 오탐을 줄인 프롬프트를 공개했습니다. 무료 샘플도 이메일 없이 받을 수 있습니다.

링크: https://harry-kim-droid.github.io/am_ps_method/

비슷하게 AI 리뷰를 운영하고 있다면 어떤 지표로 품질을 판단하는지 궁금합니다.

## OKKY용

### 제목

1인 프로젝트에 Codex PR 리뷰 자동화를 붙이며 정리한 것들

### 본문

혼자 만드는 프로젝트에서 테스트 누락과 배포 위험을 한 번 더 확인하려고 Codex PR 리뷰 워크플로를 만들었습니다.

핵심은 자동 수정이 아니라 읽기 전용 검토였습니다. `contents: read`와 read-only sandbox로 시작하고, 파일·영향·수정 방향·테스트가 없는 지적은 버리도록 프롬프트를 좁혔습니다.

설치 방법, 비용 관리, 주간 기술부채 리포트 운영법을 글 5개로 정리했고 PR 리뷰 프롬프트는 무료로 공개했습니다.

https://harry-kim-droid.github.io/am_ps_method/

실제 프로젝트에 적용하면서 불편했던 부분이나 필요한 언어/프레임워크가 있다면 피드백 부탁드립니다.

## LinkedIn / 개인 SNS용

혼자 개발해도 코드리뷰의 “두 번째 시선”은 만들 수 있지 않을까?

GitHub PR이 열릴 때 Codex가 버그, 누락된 테스트, 배포 위험을 먼저 확인하는 읽기 전용 자동화 키트를 만들었습니다. AI가 코드를 바로 고치게 하지 않고, 사람이 병합 전에 판단할 근거를 만드는 데 집중했습니다.

무료 PR 리뷰 프롬프트와 실제 설정 과정을 공개합니다.
