기술부채 목록이 길어질수록 실제 행동은 줄어듭니다. 주간 리포트의 목적은 모든 문제를 수집하는 것이 아니라 다음 한 주에 처리할 항목을 고르는 것입니다.

## 최근 변경부터 보기

오래된 전체 코드보다 최근 변경 파일, 새 TODO, 반복된 리뷰 지적, 실패한 테스트를 먼저 살핍니다. 지금 활발히 바뀌는 영역이 사용자 위험과 연결될 가능성이 높습니다.

## 다섯 개로 제한하기

각 항목에는 근거 파일, 방치했을 때 영향, 예상 작업 크기, 이번 주에 할 최소 조치를 적습니다. 상위 다섯 개만 본문에 두고 나머지는 보류 목록으로 보냅니다.

## 리포트를 작업으로 연결하기

매주 한 항목 이상을 GitHub Issue로 바꾸고 다음 리포트에서 완료 여부를 확인합니다. 자동 생성 파일만 쌓이고 실제 정리가 없다면 리포트 빈도나 범위를 줄여야 합니다.
